API & Integration

All forecast data is served as static JSON and CSV files — no API key required. Files refresh automatically every 15 minutes.

Endpoints

Every zone/provider combination exposes three files:

{ORIGIN}/{ZONE}/{PROVIDER}/tariff.json   # evcc forecast JSON
{ORIGIN}/{ZONE}/{PROVIDER}/tariff.csv    # flat CSV with forecast bounds
{ORIGIN}/{ZONE}/{PROVIDER}/chart.png     # uncertainty chart

Global metadata and model quality:

{ORIGIN}/api/meta.json     # zone & provider registry
{ORIGIN}/api/quality.json  # model quality metrics

evcc integration

Use the forecast source type (requires evcc ≥ 0.130). Replace NL/wholesale with your zone and provider — see the provider page for the exact URL. 

forecast:
  source: http
  uri: {ORIGIN}/NL/wholesale/tariff.json
  cache: 15m
  jq: '.rates | tostring'

The JSON uses type: forecast with a rates[] array. Each entry has start, end (UTC, Z-suffix) and value (all-in EUR/kWh).

JSON schema

{
  "type": "forecast",
  "provider": "wholesale",
  "zone": "NL",
  "currency": "EUR",
  "unit": "kWh",
  "generated_at": "2026-03-25T09:00:00Z",
  "model_version": "202603250900",
  "rates": [
    {
      "start": "2026-03-25T00:00:00Z",
      "end":   "2026-03-25T01:00:00Z",
      "value": 0.1234,
      "is_actual": true
    },
    ...
  ]
}

is_actual: true when the hour uses a confirmed ENTSO-E day-ahead price; false for model-predicted slots.

Python / pandas

import pandas as pd

# Flat CSV — includes wholesale forecast bounds
df = pd.read_csv(
    "{ORIGIN}/NL/wholesale/tariff.csv",
    parse_dates=["start", "end"],

Time resolution — why hourly?

The forecast uses one-hour slots, even though the underlying market now clears at 15-minute resolution. This is a deliberate design choice grounded in what the underlying data and models can actually support.

The market switched to 15-minute resolution in October 2025

On 30 September 2025, EPEX SPOT switched the Single Day-Ahead Coupling (SDAC) auction from 60-minute to 15-minute Market Time Units (MTU) in all participating bidding zones simultaneously (Ireland remains at 30-minute). This transition is mandated by EU Regulation 2019/943, which requires all TSOs and NEMOs to adopt 15-minute imbalance settlement across European coupled markets. As a result, ENTSO-E's Transparency Platform now publishes 15-minute day-ahead clearing prices for all affected zones.

Alongside the 15-minute prices, EPEX SPOT also publishes a 60-minute price index defined as the strict arithmetic average of the four corresponding quarter-hour clearing prices. This hourly index is what the model currently trains on. (Source: EPEX SPOT)

Why not forecast at 15-minute resolution?

Forecasting the four individual quarter-hour prices within each hour is a meaningfully different — and harder — problem than forecasting the hourly average. The intra-hour spread between quarter-prices is driven primarily by real-time balancing dynamics: short-notice renewable deviations, last-minute demand shifts, and transmission constraints that resolve within minutes. These signals are not available in day-ahead input features such as weather forecasts, solar and wind output estimates, or calendar data. A model trained on those features cannot reliably predict how prices will differ between 14:00–14:15 and 14:45–15:00; it can only predict the mean.

Input features don't support finer resolution

Every input to the model carries meaningful signal at the hourly level. Numerical weather prediction ensembles, solar irradiance estimates, and wind power output are all aggregated over hours by the time they are usable as features. Splitting them into 15-minute blocks produces four identical (or linearly interpolated) values per hour — not independent information. Quadrupling the output space without a proportional increase in informative features increases model complexity for no accuracy benefit, while making the training target noisier and harder to learn.

Forecast uncertainty already dominates

Beyond the first 24 hours, prediction intervals on any day-ahead price forecast widen substantially. By day 3–9, the uncertainty band routinely spans several cents per kWh. Within that range, the typical intra-hour spread — a few tenths of a cent on most days — is completely swamped by forecast error. Reporting 15-minute slots would add four numbers per hour that are statistically indistinguishable from each other.

Hourly is sufficient for scheduling decisions

Home automation use cases — EV charging, heat pump scheduling, battery dispatch — optimise across hours, not quarter-hours. No actionable decision changes between adjacent 15-minute slots in a plan made the evening before. evcc's scheduler itself aligns to hourly blocks when planning ahead, making finer granularity in the forecast redundant in practice.

Ongoing evaluation

As the model continues to develop, the question of 15-minute slots is actively re-evaluated. There is currently no strong reason to introduce them: the arguments above hold, and adding resolution that the model cannot meaningfully exploit would reduce rather than improve forecast quality. That said, future advances — better sub-hourly feature engineering, richer intraday training data, or demonstrated scheduling benefit — could change that assessment. The decision will be revisited whenever there is concrete evidence that 15-minute forecasts would provide real value to users.
